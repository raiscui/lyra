## 两个方向(至少两个可选方案)

1. 不惜代价,最佳方案(相机+高斯联合):
   - 两阶段: 先 pose-only(固定高斯),再 joint(小 lr 微调高斯).
   - 强约束: pose delta 的 L2 + smoothness + anchor(第一帧固定).
   - 损失: RGB Charbonnier/L1 + SSIM(可选) + 高斯正则(尺度/opacity).
2. 先能用,后面再优雅(只优化高斯):
   - 相机完全不动,只做 gaussians 的短迭代 refinement.
   - 更容易稳定收敛,也更符合"轨迹必须严格一致"的要求.

# 任务计划: 探索 `sample.py` 链路下超分视频对高斯清晰度的作用边界

## 目标

在不改代码的前提下,确认两件事:

1. 当前 `accelerate launch sample.py` 流程里,单纯提高输入视频分辨率,是否能直接改善高斯输出清晰度.
2. 如果主链路不适合这样做,是否可以在 `sample.py` 结束后,再用“内容完全相同但已超分”的对等视频做一轮高斯后处理优化.

## 阶段

- [x] 阶段1: 回读历史结论与主链约束
- [x] 阶段2: 追踪 `sample.py` / provider / dataset 的真实输入依赖
- [x] 阶段3: 评估“直接提分辨率”与“末尾再优化”两类方案
- [x] 阶段4: 形成建议与后续落地方向

## 关键问题

1. `sample.py` 是优先读 `rgb_latents` 还是优先读 `rgb mp4`?
2. `intrinsics` 是否会跟随裁剪缩放一起变化?
3. “超分后再训练高斯”更像主链改造,还是更像 per-scene refinement?

## 两个方向

1. 不惜代价,最佳方案:
   - 不把超分视频硬塞回现有 feed-forward 主链.
   - 保持 `sample.py` 作为初始化器.
   - 在末尾追加一个独立的 scene-level refinement 阶段,用超分后的对等视频做高分辨率监督,优化高斯参数.
2. 先能用,后面再优雅:
   - 先不碰高分辨率 supervision.
   - 继续沿当前 `gaussian_scale_cap` 和鲁棒 refinement V2 的方向做 704x1280 档位的去重影.

## 做出的判断

- 判断: 在当前默认配置下,单纯“把视频变成更高分辨率”并不能自然传导成更清晰的高斯.
- 原因:
  1. `sample.py` 在有 `rgb_latents` 时直接使用 latent.
  2. 默认 demo 配置 `load_latents: true`,且对应 dataset `has_latents: true`.
  3. provider 会把 RGB 与 intrinsics 一起裁剪缩放到 `img_size`.
  4. 当前 demo / 训练主档位固定在 `img_size: [704, 1280]`.
- 判断: “在最后追加基于超分对等视频的高斯 refinement”在架构上是可行且更合理的路线.
- 条件:
  1. 超分视频必须和原视频内容、时序、裁剪完全对齐.
  2. refinement 需要按超分倍率同步缩放 intrinsics.
  3. 更适合做 `gaussian-only refinement`,而不是先碰 pose.

## 状态

**目前已完成阶段4**: 已形成结论. 主链直接提分辨率不属于当前现成可行路径. 更值得落地的是“`sample.py` 输出高斯后,再用超分对等视频做鲁棒的 scene-level gaussian refinement”.
