## [2026-03-09 14:25:00 UTC] 问题: FlashVSR-Pro 在 A800 主机上首次无法直接运行

### 现象
- `infer.py` 初始报错缺少 Python 依赖:
  - `imageio`
  - `ftfy`
  - `datasets`
  - `ffmpeg-python`
  - `pytorch-lightning`
  - `torchsde`
- `Block-Sparse-Attention` 初次编译失败:
  - `nvcc fatal: Unsupported gpu architecture 'compute_120'`
- 编译后又遇到动态库加载问题:
  - `ImportError: libc10.so: cannot open shared object file`

### 原因
- 当前 `pixi` 环境并未默认包含 FlashVSR-Pro 全套依赖
- `Block-Sparse-Attention/setup.py` 默认架构列表包含 `120`, 但当前 nvcc 不支持
- `torch/lib` 未自动出现在本地 runner 的动态库搜索路径中

### 修复
- 在 `pixi` 环境补装 FlashVSR-Pro 缺失依赖
- 编译 `block_sparse_attn` 时显式设置:
  - `BLOCK_SPARSE_ATTN_CUDA_ARCHS=80`
  - `MAX_JOBS=4`
- 运行 FlashVSR-Pro 时显式设置:
  - `LD_LIBRARY_PATH=/workspace/lyra/.pixi/envs/default/lib/python3.10/site-packages/torch/lib:/workspace/lyra/.pixi/envs/default/lib:$LD_LIBRARY_PATH`
  - `PYTHONPATH=/workspace/FlashVSR-Pro:/workspace/lyra`

### 验证
- `env ... pixi run python3 /workspace/FlashVSR-Pro/infer.py --help`
  - 退出码 `0`
- `scripts/run_flashvsr_reference.py --view-ids 5 --scene-stem 00172`
  - `status=succeeded`
- `scripts/run_flashvsr_reference.py --view-ids 0,1,2,3,4 --scene-stem 00172`
  - `5` 路全部 `status=succeeded`

### 结论
- 这台 A800 主机可以本地重新生成 FlashVSR-Pro reference
- 后续若重跑,优先复用上述环境变量与架构限制,避免再次踩到 `compute_120` 与 `libc10.so` 问题

## [2026-03-09 16:45:00 UTC] 问题: full-view `SR -> Stage 2B` warm-start 在 CUDA scene 下因 mixed-device patch intrinsics 崩溃

### 现象
- 真实命令使用:
  - `--start-stage stage2b`
  - `--resume`
  - patch supervision 开启
  - `reference-mode super_resolved`
- 运行在 `bootstrap_stage2b_from_current_gaussians()` 时崩溃:
  - `RuntimeError: Expected all tensors to be on the same device, but found at least two devices, cuda:0 and cpu!`
- 栈定位到:
  - `src/refinement_v2/runner.py: build_patch_intrinsics`

### 原因
- warm-start 路径里的 evaluation render 会先把 `pred_rgb / residual_map` 落到 CPU
- 于是 `patch_windows_ref` 留在 CPU
- 但主场景 `self.scene.intrinsics_ref` 已在 CUDA
- `build_patch_intrinsics()` 之前只写了:
  - `to(dtype=patch_intrinsics.dtype)`
  - 没有把 patch window offset 迁到同一 device

### 修复
- 在 `build_patch_intrinsics()` 中显式把 `patch_windows_ref` 对齐到:
  - `device=patch_intrinsics.device`
  - `dtype=patch_intrinsics.dtype`
- 新增 CUDA 回归测试:
  - `test_render_patch_prediction_aligns_cpu_patch_windows_with_cuda_scene`

### 验证
- `PYTHONPATH="$(pwd)" pixi run pytest -q tests/refinement_v2/test_patch_supervision.py -k aligns_cpu_patch_windows_with_cuda_scene`
  - 通过
- `PYTHONPATH="$(pwd)" pixi run pytest -q tests/refinement_v2/test_runner_stage2b.py tests/refinement_v2/test_patch_supervision.py -k 'stage2b or render_patch_prediction'`
  - 通过

### 结论
- `SR -> Stage 2B` warm-start 的当前 blocker 已解除
- 后续可继续回到真实 full-view 命令验证指标收益

## [2026-03-09 16:50:00 UTC] 问题: warm-start 恢复后缺失 `sr_selection_map` 时, patch weight fallback 再次触发 mixed-device 崩溃

### 现象
- 修完 `build_patch_intrinsics()` 后, full-view `SR -> Stage 2B` 真实命令继续前进
- 但随后又在:
  - `weight_builder.combine_sr_weights`
  崩溃
- 报错仍是:
  - `Expected all tensors to be on the same device, but found at least two devices, cuda:0 and cpu!`

### 原因
- `_compute_patch_losses()` 中:
  - `robust_patch` 会在组合前迁到 `pred_patch.device`
- 但当 `sr_selection_map` 为 `None` 时:
  - fallback 分支直接 `torch.ones_like(robust_patch)`
  - 当时的 `robust_patch` 仍是 CPU tensor
- 因而 `w_robust` 在 CUDA, `w_sr_select` 在 CPU, 最终在 `combine_sr_weights()` 再次炸掉

### 修复
- 提前把 `robust_patch` 统一迁到 `pred_patch.device/pred_patch.dtype`
- `sr_selection_map is None` 时, fallback 改为在 `pred_patch.device` 上构造 `ones_like`
- 新增 CUDA 回归测试:
  - `test_bootstrap_stage2b_with_patch_supervision_handles_missing_sr_selection_on_cuda`

### 验证
