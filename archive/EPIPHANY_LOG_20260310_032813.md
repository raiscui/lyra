## [2026-03-09 14:25:00 UTC] 主题: FlashVSR-Pro 在 A800 上的真正脆弱点不是模型,而是 Block-Sparse-Attention 架构列表与运行时库路径

### 发现来源
- 本轮为修正 external SR reference 口径,在当前 A800 主机上重新部署并运行 `FlashVSR-Pro`

### 核心问题
- 表面看像是“FlashVSR 跑不起来”,但真正的高频绊脚石有两个:
  - `Block-Sparse-Attention/setup.py` 默认会带上 `120` 架构
  - 本地 runner 不会自动继承到足够的 `torch/lib` 动态库路径

### 为什么重要
- 如果不提前记住这两个点, 每次换机器或重开环境,都可能重新踩进:
  - `nvcc fatal: Unsupported gpu architecture 'compute_120'`
  - `ImportError: libc10.so`
- 这会让团队误以为是模型权重、repo 或 CUDA 本身坏了, 从而把时间浪费在错误方向上

### 当前结论
- 对当前 A800 / CUDA 12.6 / torch 2.6+cu124 组合:
  - 编译时优先使用 `BLOCK_SPARSE_ATTN_CUDA_ARCHS=80`
  - 运行时显式补 `LD_LIBRARY_PATH=torch/lib`
- 一旦这两点到位, `FlashVSR-Pro` 本身可以稳定跑通

### 后续讨论入口
- 如果后续需要把 FlashVSR 纳入固定流程,优先把这两条环境约束封装进脚本或文档,而不是继续依赖人工记忆

## [2026-03-09 17:04:00 UTC] 主题: warm-start resume 不能假设 `sr_selection_map` 这类运行期缓存一定会被恢复

### 发现来源
- 本轮继续执行 full-view `SR stage3sr -> Stage 2B` warm-start 时, 真实命令先后暴露两处 mixed-device 崩溃

### 核心问题
- `restore_latest_state()` 当前只恢复:
  - `gaussians`
  - `diagnostics_state`
  - `pose_delta`
- 它不会恢复 `sr_selection_map` 这类运行期张量缓存
- 如果 warm-start 路径仍默认把这些缓存当成“已经在内存里”, 很容易在 patch supervision fallback 分支里炸出新的设备问题

### 为什么重要
- 这不是单点 bug
- 它揭示的是 resume 语义边界:
  - 哪些是 durable state
  - 哪些只是本轮临时缓存
- 后续如果还给 refinement 加新的 dense cache / selection map / guidance tensor, 都会遇到同一类问题

### 当前结论
- warm-start 可依赖的只有显式持久化状态
- 对未持久化的运行期缓存:
  - 要么重新构建
  - 要么提供设备一致的安全 fallback
- 不能把“上一轮 enhanced 路径跑过”误写成“这一轮 resume 后缓存一定还在”

### 后续讨论入口
- 以后继续加 resume 相关功能时, 先检查:
  - 这个张量有没有被持久化
  - 如果没有, fallback 是否也满足 device/dtype 一致性
