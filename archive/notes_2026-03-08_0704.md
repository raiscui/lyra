## 2026-03-06 07:28 UTC: `SplatSuRe` / `Mip-Splatting` / `EDGS` 进一步映射到现有 spec

### 结论摘要

- `SplatSuRe` 不是普通参考论文,而是当前“超分对等视频如何安全参与监督”的最直接增强项.
- `Mip-Splatting` 的 `3D smoothing` 值得吸收,但不能直接替换当前压大尺度尾部的 `scale_reg`.
- `EDGS` 当前不适合进入主线,更适合保留成未来的局部 reinit fallback.

### 对当前 spec 的直接影响

- `SR patch supervision` 升级为 `selective SR patch supervision`.
- 新增三个核心对象:
  - `gaussian_fidelity_score`
  - `W_sr_select`
  - `W_final_sr = W_robust * W_sr_select`
- loss 侧从单一尺度项升级成两条互补约束:
  - `L_scale_ceiling`
  - `L_sampling_smooth`

### 为什么 `Mip` 不能直接替换旧 `scale_reg`

- 当前 `scale_reg` 主要负责压大尺度尾部:
  - 去雾感
  - 去厚表面
  - 去叠层
- `Mip-Splatting` 的 `3D smoothing` 更偏:
  - 抑制不受支持的高频小 Gaussian
  - 防止提高监督分辨率后出现 aliasing-driven 伪细节
- 所以合理关系不是“替换”,而是“拆成两条互补约束”.

### 为什么 `EDGS` 只保留为 deferred idea

- `EDGS` 的本质是 better initialization module.
- 当前 Lyra 主线不是“重新初始化一个 3DGS”,而是“对 feed-forward gaussians 做后置 refinement”.
- 如果当前就引入 `EDGS`,会把主线从“后置优化”带回“重做初始化”.
- 更合理的保存方式是:
  - `EDGS-style local reinitialization`
  - 只有在 Stage 3A + Stage 3SR 后仍出现结构缺失时,才考虑研究.

### 真实验证中暴露并修正的关键问题

1. `weight_map` 通过 EMA 复用时保留上一轮计算图.
   - 现象: `stage2a` 第二轮开始报 `Trying to backward through the graph a second time`.
   - 修正: `WeightBuilder.build_weight_map()` 显式对 `residual_map` 与 `prev_weight_map` 做 `detach`, 返回值也保持 detached.
   - 结论: robust weight 应作为静态监督权重,不应参与跨 iter 反传.

2. refinement 之前只读取 demo 顶层 YAML,没有按 `config_path` 合并训练配置链.
   - 现象: 真实 dataloader 构建时报 `Missing key data_mode`.
   - 修正: `data_loader._load_scene_config()` 现在会解析并合并 `config_path` 链,再叠加 refinement 覆盖项.
   - 结论: refinement 不能把 demo config 当成完整训练 config 用.

3. 真实 demo 资产与 demo YAML 的 `dataset_name` 不一定一致.
   - 现象: 当前仓库只有 `lyra_static_demo_generated`, 但 `configs/demo/lyra_static.yaml` 指向 `lyra_static_demo_generated_one`.
   - 修正: CLI 新增 `--dataset-name`, 允许 refinement 在不改全局 demo YAML 的前提下切到本机真实存在的数据资产.
   - 结论: refinement 入口要对本地资产差异更宽容.

4. provider 会默认读 `depth` / `latents`,并在缺文件时无限重试.
   - 现象: 真实运行时卡在 dataloader,反复尝试读取不存在的 `depth/*.zip`.
   - 修正: refinement 专用 loader 覆盖 `use_depth=False`、`load_latents=False`,并补上 `target_index_manual=None` 默认值.
   - 结论: refinement 只需要 `RGB + cam_view + intrinsics`,不该继承训练链路里的所有 IO 依赖.

## 2026-03-06 08:13 UTC: `opacity/pruning` 最小可落地设计判断

### 目标

- 不改动现有 `refinement_v2` 主线入口与默认行为.
- 通过 `enable_pruning` 显式开关启用 pruning.
- 先做 `stage2a` 内的 low-opacity pruning, 不提前混入 patch supervision 或几何大改.

### 设计选择

1. 配置层
   - `RefinementRunConfig` 增加 `enable_pruning`.
   - `StageHyperParams` 增加:
     - `opacity_prune_threshold`
     - `prune_every`
     - `prune_warmup_iters`
     - `prune_max_fraction`
     - `min_gaussians_to_keep`

2. 控制层
   - `StageController.should_prune_now(iteration)` 采用 1-based iteration.
   - 规则:
     - pruning 开关未开 -> 不 prune
     - `iteration <= prune_warmup_iters` -> 不 prune
     - `prune_every <= 0` -> 不 prune
     - 其余按 `(iteration - prune_warmup_iters) % prune_every == 0` 触发

3. 高斯适配层
   - `collect_prune_candidates()` 只负责找出低 opacity 候选.
   - `prune_low_opacity()` 负责:
     - 按 opacity 从低到高裁掉
     - 受 `prune_max_fraction` 和 `min_gaussians_to_keep` 双重限制
     - 同步裁剪 `means/opacity/scales/rotations/colors/initial_means`

4. runner 接线
   - 先保持现有 loss / render / clamp 主流程不变.
   - 在单轮 step 之后触发 pruning.
   - 一旦实际 prune 成功,立即重建 optimizer.
   - prune 结果单独写 `pruning/*.json`, stage metrics 在下一轮自然反映新高斯统计.

### 风险判断

- 如果把 pruning 放在默认路径且默认开启,会改变现有真实验证基线,不适合.
- 如果 prune 后不重建 optimizer,Adam 内部 state 会继续引用旧参数,行为不可靠.
- 如果不裁 `initial_means`,后续 `stage2b/phase4` 的 `means_delta_cap` 会和参数数量错位.

## 2026-03-06 08:46 UTC: `patch-based supervision` 最小实现判断

### 当前可直接复用的基础

- `SceneBundle` 已经统一承载 `gt_images/cam_view/intrinsics`.
- `RefinementRunner` 已有统一的 `render_current_scene()` 和 `stage2a` 主循环.
- `src/rendering/gs_deferred_patch.py` 已经说明 patch 渲染的关键相机语义:
  - patch 内保持 `fx/fy` 不变
  - 通过 `cx/cy` 减去 patch 左上角偏移实现局部渲染相机

### 最小 MVP 判断

1. 先不直接接真实 SR 视频文件.
   - 先把 `SceneBundle` 扩成可携带 `reference_images` 与 `intrinsics_ref`.
   - 默认 `reference_images = gt_images`, `intrinsics_ref = intrinsics`, `reference_mode = native`.
   - 这样 native patch path 先成立,后面再接 SR 参考图不会推翻接口.

2. patch 路径先做成“参考 patch 抽样 + patch loss”.
   - MVP 先不强依赖 `DeferredBPPatch` 真正执行 patch-only rasterization.
   - 先从当前全帧渲染结果里裁 patch,但 patch 相机语义与 reference 提取逻辑要完整.
   - 这样可以先稳定契约和 loss,再决定下一轮是否接真正的 patch renderer.

3. 需要显式记录两套 intrinsics.
   - `intrinsics`: native/full-frame 渲染使用.
   - `intrinsics_ref`: reference patch 监督使用.
   - 对 SR 模式, `intrinsics_ref` 必须按分辨率倍率缩放.

4. runner 中至少补三个原子函数.
   - `sample_patch_windows(...)`
   - `gather_reference_patch(...)`
   - `build_patch_intrinsics(...)`

### 风险判断

- 如果现在把 patch 渲染直接强绑 `gsplat` patch rasterizer,测试会变重,还会扩大引入范围.
- 如果不先把 `intrinsics_ref` 独立出来,后面一接 SR 就会重新拆接口.
- 因此本轮最稳妥的做法是:
  - 先固化 patch supervision contract
  - 先在 runner 内跑通 patch sampling + patch loss
  - 再决定是否把 patch-only renderer 作为下一轮增强

    - 根据 external reference 分辨率自动推断 `sr_scale`
    - 把 external reference 真正放进 `SceneBundle.reference_images`
- 因此 `Long-LRM` 线已经不再被“外部 SR/reference 根本喂不进来”这个问题阻塞。

### 现在这条线真正的 critical path

#### 1. 不要再优先碰通用基础设施

- `W_robust`
- baseline diagnostics
- Stage 3A 等价主线
- opacity/pruning
- external reference 接入
这些共享基础设施已经足够支撑下一阶段了。

#### 2. 现在最该做的是把“普通 patch supervision”升级成“selective SR”

- 当前 runner 里 `_compute_patch_losses(...)` 还直接给 patch weight 全 1。
- 所以现在 external reference 虽然能进来,但还是“谁有 patch 就吃谁”,还不是 spec 里的:
  - `gaussian_fidelity_score`
  - `W_sr_select`
  - `W_final_sr = W_robust * W_sr_select`

#### 3. 最稳的推进顺序应改成 4 步

1. 先做一个“无行为变化”的阶段拆分.
   - 把当前 `run_stage2a()` 里混着的 patch supervision 抽出来。
   - 先让代码结构上显式出现:
     - Stage 3A native cleanup
     - Phase 3S fidelity / selection
     - Stage 3SR selective SR
   - 这一步先不追求效果提升,只追求阶段边界清晰。

2. 再做 renderer info 上抬.
   - `src/rendering/gs.py` 已经拿到 `rasterization(...).info`.
   - 但现在没往 refinement 层传。
   - `gaussian_fidelity_score` 最自然的实现入口就在这里。
   - 第一版先不要急着定最终公式,而是先把 `info` 结构暴露出来,锁住 keys / shapes / test contract。

3. 然后实现 `W_sr_select` / `W_final_sr`.
   - 先在 `WeightBuilder` 里补两类接口:
     - `build_sr_selection_weight(...)`
     - `combine_sr_weights(...)`
   - 第一版即使只做到 patch-level 或 view-level selection,也比全 1 patch weights 前进一大步。

4. 最后补 `L_sampling_smooth`.
   - 它现在应该作为 selective SR 之后的保护项进入主线。
   - 语义是“限制不受支持的高频”,不是替代 `L_scale_ceiling`。

### 为什么 Stage 3B 不该现在先做

- Stage 3B 的共享代码主体已经存在。
- 但 `Long-LRM` 规格要求它发生在:
  - Stage 3A 已跑完
  - Stage 3SR 已跑完
  - selective SR 仍然没有解决残余局部结构问题
- 也就是说,对这条线来说,Stage 3B 现在不是实现瓶颈,而是 gating 语义还没有接上前面的 selective SR 阶段。

## 2026-03-06 11:52 UTC: 用户确认的 `SplatSuRe + Mip-Splatting` 方案约束

### `SplatSuRe`

- 值得直接学习和吸收。
- 当前最值得借的不是整套论文流程,而是 3 个核心对象:
  1. per-Gaussian fidelity score
  2. per-view SR weight map
  3. LR + SR 的 selective joint objective
- 它对当前路线的最大价值,就是把“超分后的对等视频”从危险的全图监督,变成可控的局部高频监督。
- 这条线的主要风险也被明确记录:
  - fidelity 依赖 Gaussian 的跨视图 footprint / support 统计
  - 如果初始 gaussians 太脏, fidelity score 会被污染
- 因此顺序必须保持:
  1. 先 Stage 3A 清 opacity / color
  2. 再上 `SplatSuRe-style selective SR`

### `Mip-Splatting`

- 值得参考,但不应抢成当前主线。
- 当前最值得借的不是完整 renderer-level 方案,而是:
  1. 3D smoothing filter 背后的 sampling-frequency constraint 思想
  2. alias-free supervision / rendering 的目标意识
- 它对当前路线的最大价值,是在 reference supervision 拉高到 HR 时:
  - 约束 unsupported 高频
  - 减少 HR render aliasing
- 同时明确:
  - 它不该被期待单独解决:
    - 厚表面
    - 双轮廓
    - SR 假细节多视图不一致
- 更准确的定位是:
  - 它负责“如何更稳地渲染和约束”
  - 而不是“该不该信这些 SR 细节”

### 当前方案应保持的关键关系

- `SplatSuRe` 负责“选择哪些 SR 细节值得信”。
- `Mip-Splatting` 负责“在更高分监督下如何减少不被支持的 alias / 高频伪细节”。
- 因此现在的正确顺序是:
  1. 先把 `SplatSuRe-style selective SR` 并入主线。
  2. 再把 `Mip-inspired 3D smoothing` 作为互补约束并入。
  3. 不把 `Mip` 误用成 selective SR 的替代。
  4. 不把 `Mip` 设计成替换当前 `scale_reg` 的唯一手段,除非后续实验证明旧项确实应被移除。
