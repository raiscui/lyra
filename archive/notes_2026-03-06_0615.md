# 笔记: `megatron.core` 导入失败的根因分析

## 背景与复现

- `scripts/test_environment.py` 只做一件事: `importlib.import_module()` 逐个导入关键包.
- `[ERROR] megatron.core` 的含义很直接: `import megatron.core` 在导入过程中抛异常.
- 这个脚本不会打印异常栈,所以需要单独 `import megatron.core` 才能看到真正原因.

## 关键发现

### 发现1: 仓库里确实依赖 Megatron Core

- `requirements_gen3c.txt` 包含 `megatron-core==0.10.0`.
- 仓库代码 `cosmos_predict1/` 多处直接 `from megatron.core import ...`.
  - 因此,只要你运行这些模块,`megatron.core` 不能导入就会直接崩.

### 发现2: 失败不是 "没装",而是导入链路被 Transformer Engine 卡死

在 `.pixi/envs/default` 环境中:

- `megatron-core` 已安装.
- 但 `megatron.core` 的导入链路里会尝试探测 TE 的 `Float8Tensor`.
- 正常情况下:
  - TE 不存在 -> 抛 `ModuleNotFoundError`,并被上层捕获,导入继续.
  - TE 存在且正确 -> 正常导入.
- 异常情况(本次):
  - TE "半安装": 只有 `transformer_engine` / `transformer_engine_cu12`,缺少 PyTorch 扩展 `transformer_engine_torch`.
  - 这会导致 `import transformer_engine.pytorch` 抛 `StopIteration`.
  - 而 Megatron Core 侧只捕获 `ImportError/ModuleNotFoundError`.
  - 结果: `StopIteration` 泄漏,直接把 `import megatron.core` 弄挂了.

## 修复思路

### 方案A(最佳方案,性能优先)

把 TE 装完整:

- 保证三件套版本一致:
  - `transformer_engine==1.12.0`
  - `transformer_engine_cu12==1.12.0`
  - `transformer_engine_torch==1.12.0`
- 优先用预编译 wheel(最快,最稳定).
- 如果只能源码编译:
  - 需要系统编译工具链(build-essential).
  - 需要 CUDA include/lib 路径.
  - 需要 cuDNN 头文件(`cudnn.h`).
  - 需要 NVTX 头文件(`nvtx3/nvToolsExt.h`).

### 方案B(先能用,后面再优雅)

如果你不需要 TE 加速:

- 直接卸载 `transformer_engine` 和 `transformer_engine_cu12`.
- 让 Megatron Core 的 TE 探测走 `ModuleNotFoundError` 分支,不再被 `StopIteration` 卡死.
- 这样通常也能让 `megatron.core` 正常导入.

## 本次采取的措施(事实记录)

- 目标环境: `.pixi/envs/default/bin/python`.
- 已通过源码编译安装 `transformer_engine_torch==1.12.0`.
- 随后验证:
  - `import transformer_engine.pytorch` 成功.
  - `import megatron.core` 成功.
  - `scripts/test_environment.py` 全部 `[SUCCESS]`.

# 笔记: 切换 T5 prompt encoder 到 ModelScope `mindnlp/t5-11b`

## 结论先行

在本项目里,`t5-11b` 的职责是"把 prompt 文本编码成 embedding".
它不是用来做文本生成的.

## 代码锚点(事实)

- 加载入口:
  - `cosmos_predict1/utils/base_world_generation_pipeline.py` 的 `_load_text_encoder_model()`.
- 编码实现:
  - `cosmos_predict1/auxiliary/t5_text_encoder.py` 使用 `T5TokenizerFast` + `T5EncoderModel`.
- 下游硬约束(形状相关):
  - token 数上限: `cosmos_predict1/autoregressive/training/networks/transformer.py` 里 `_T5_NUM_TOKENS = 512`.
  - embedding 维度: `cosmos_predict1/autoregressive/configs/base/model.py` 里 `context_dim` 默认 `1024`.
  - `DummyT5TextEncoder` 也直接构造了 `(batch, max_length, 1024)` 的占位 embedding.

## 为什么不能"随便换"

就算新模型也叫 T5,只要 embedding 维度不一致,主模型的 cross-attention 就会直接 shape mismatch.
就算维度一致,如果 embedding 分布差异很大,效果也可能掉很多.

所以本次把 `google-t5/t5-11b` 换成 `mindnlp/t5-11b`,更像是"换权重来源/镜像".
目标是让下载更稳定,并减少对 Hugging Face 的依赖.

# 笔记: 纠正 T5 prompt encoder 误切换(还原为 Hugging Face `google-t5/t5-11b`)

## 背景

用户明确指出:

- `mindnlp/t5-11b` 不是 Hugging Face 的 `google-t5/t5-11b`.
- 继续沿用会导致效果与行为不可控,并且当前目录结构也无法被 `transformers` 正常加载.

## 直接证据(现象层)

运行推理脚本时,`transformers` 报错:

- 本地目录缺少 `pytorch_model.bin` / `model.safetensors` 等权重文件.

这说明当前落盘的 `checkpoints/mindnlp/t5-11b` 不符合 `from_pretrained(<local_dir>)` 的预期结构.

## 结论(本质层)

在本项目里:

- T5 prompt encoder 需要的是 `transformers` 能直接加载的 HF 格式权重.
- 因此必须回到 `google-t5/t5-11b`,并且只从本地固定目录加载,避免再发生"名字像但不是同一个模型"的误用.

## 新的约定(可操作)

- 默认加载路径固定为: `/model/HuggingFace/google-t5/t5-11b`.
- 如果目录不存在或不完整:
  - 直接报错提示用户从 Hugging Face 准备该目录.
  - 不再自动回退到其它目录,避免静默用错模型.

# 笔记: `gen3c_single_image_sdg.py` 视频生成是否可替换,以及产物依赖关系

## 时间

- 2026-03-04 13:32 UTC

## 目标(对应用户问题)

- 梳理 `cosmos_predict1/diffusion/inference/gen3c_single_image_sdg.py` 的执行流程.
- 明确它生成的"视频"是否必须由项目内的 Gen3C 视频扩散模型产生.
- 梳理与视频同时落盘的其他产物,它们和视频的先后/依赖关系.
- 评估: 如果改为载入外部工具生成的视频,能否继续生成其它产物.

## 关键源码位置(带结论)

### 1) 这个脚本实际落盘的产物有哪些?

脚本在同一个 `args.video_save_folder` 下写入 4 类产物(同一个 `clip_name` 前缀):

- RGB 视频: `rgb/{clip_name}.mp4`
- 相机位姿: `pose/{clip_name}.npz` (c2w, 4x4)
- 相机内参: `intrinsics/{clip_name}.npz` (fx, fy, cx, cy)
- 视频 latent: `latent/{clip_name}.pkl`

对应代码:

- 生成视频(并返回 latent): `pipeline.generate(..., return_latents=True)` 在 `gen3c_single_image_sdg.py` 约第 385-398 行.
- 保存 pose/intrinsics/latent/rgb: `gen3c_single_image_sdg.py` 约第 490-558 行.

### 2) 这些产物在下游是否真的会被用到?

`src/models/data/radym.py` 明确按固定目录结构读取:

- `latent/{data_key}.pkl` -> `DataField.LATENT_RGB`
- `pose/{data_key}.npz` -> `DataField.CAMERA_C2W_TRANSFORM`
- `intrinsics/{data_key}.npz` -> `DataField.CAMERA_INTRINSICS`

这意味着: 如果你想把生成的视频喂给后续 3D 重建(例如 `sample.py` 所用的数据流),仅有 mp4 一般不够,还需要 pose/intrinsics/latent 至少其中一部分.

### 3) "其他产物"是先有视频再生成,还是伴生的?

从脚本逻辑看,它不是一个"先生成视频文件 -> 再离线计算其它产物"的两段式流水线.
更接近下面这种关系:

- pose/intrinsics: 在视频生成前,就已经通过 `generate_camera_trajectory(...)` 生成出来(但文件是在生成完后才写盘).
- latent: 与视频在同一次扩散采样中产生.脚本拿到 `pipeline.generate` 的返回值 `(video, prompt, latents)` 后,直接把 `latents` 落盘.

因此:

- pose/intrinsics 不是从视频"推导"出来的,而是脚本自己"设定"出来的相机轨迹.
- latent 也不是从 mp4 里"再计算"出来的,它是扩散模型输出的 latent(再经 tokenizer 解码得到 mp4).

### 4) latent 本质是什么? 为什么它和是否用内部视频模型有关?

在 `cosmos_predict1/diffusion/inference/gen3c_pipeline.py` 里:

- `_run_model_with_offload(..., return_latents=True)` 会在 tokenizer 解码前把 `sample` 作为 `latents` 保存.
- 随后才调用 `_run_tokenizer_decoding(sample)` 把 latent 解码成 RGB 帧.

这意味着脚本保存的 `latent/*.pkl` 对应的是"视频在 tokenizer latent 空间的表示"(并且带有 `sigma_data` 的缩放,见 `DiffusionT2WModel.encode/decode`).

## 对用户问题的直接回答(初步)

1) "视频是否必须由项目中的视频大模型生成?"
- 如果你只关心 `rgb/*.mp4` 这个视频文件: 不必须.外部工具生成的视频当然可以保存成 mp4.
- 但如果你希望下游还能用到 `latent/*.pkl` 并走默认的数据流: 你需要提供与之匹配的 latent.
  - 这件事不一定要跑"扩散生成".理论上可以用项目里的 tokenizer encoder(`model.encode`)把外部视频编码成 latent,再落盘.
  - 但这种 latent 与扩散模型直接采样出来的 latent 分布可能不同,效果是否一致需要实测.

2) "用外部工具生成视频后,是否还能生成其它产物?"
- pose/intrinsics: 可以生成,但前提是它们必须与外部视频的真实相机运动一致.否则下游 3D/渲染会被错误相机参数带偏.
- latent: 可以生成,但需要额外做"视频 -> tokenizer latent"的编码步骤(脚本当前没有这条路径).

## 潜在代码风险(后续需要确认)

- `gen3c_single_image_sdg.py` 在拼接长视频时,对 `video` 用 `np.concatenate(..., axis=0)` 很合理(按帧拼接).
  但对 `latents` 也用 `torch.cat(..., axis=0)` 进行同样的拼接,这一点从张量维度语义上看可疑.
  我建议后续用一次小规模运行打印 `latents.shape` 来确认维度约定,避免长视频时 latent 与 rgb 不一致.

# 笔记: SDG 脚本输出产物与可续跑点

## Static: `gen3c_single_image_sdg.py`

### 主要输出(每个轨迹目录)

- `pose/<clip_name>.npz`: 相机 c2w.
- `intrinsics/<clip_name>.npz`: 相机内参(fx,fy,cx,cy).
- `latent/<clip_name>.pkl`: diffusion 生成的 video latent(含 batch 维,训练侧会 `latents[0]` 去掉 batch).
- `rgb/<clip_name>.mp4`: tokenizer 解码后的像素视频.

### 可续跑关键点

- `latent` 已存在但 `rgb` 不存在: 可以跳过 diffusion,直接加载 `latent` 并用 tokenizer 解码保存 `rgb`.
- `rgb` 已存在但 `latent` 不存在: 无法反推 latent,必须重新跑 diffusion(但可选择不覆盖已有 rgb).
- `pose/intrinsics` 与 diffusion 无强依赖: 它们来自相机轨迹生成,可以独立补齐.

## Dynamic: `gen3c_dynamic_sdg.py`

输出与 Static 一致(同样四类产物). 区别在于输入数据加载来自:

- `--vipe_path`: `load_vipe_data(...)`
- 否则: `load_data_auto_detect(...)`

因此断点续跑的策略可以复用,仅"获取初始相机/深度/图像"的代码路径不同.

# 笔记: MoGe 权重下载超时与 Warp 初始化告警

## 现象拆分

1) Hugging Face `ReadTimeoutError`(HEAD):
- 触发点: `MoGeModel.from_pretrained("Ruicheng/moge-vitl")` 内部调用 `hf_hub_download(..., filename="model.pt")`.
- `huggingface_hub` 默认会对远端发起 HEAD 获取 ETag/commit,默认超时约 10s,网络慢时会出现重试日志.

2) Warp CUDA 告警:
- 触发点: `wp.init()` 初始化 CUDA 后端时,尝试获取 `cuDeviceGetUuid` 的 driver entry point,打印告警.
- 但随后仍会列出 GPU 设备并继续运行,属于"初始化阶段的兼容性告警"的概率较高.

## 关键结论

- `Ruicheng/moge-vitl` 并不是只用于 `--foreground_masking`,它还用于:
  - 单图深度估计(depth).
  - 推导初始内参(intrinsics)与初始位姿(w2c)以生成相机轨迹.
  因此在脚本当前实现下,MoGe 权重缺失时通常是"必崩",而不是"效果下降".

## 本机证据(用于判断是否影响本次运行)

- MoGe `model.pt` 已缓存在:
  - `/root/.cache/huggingface/hub/models--Ruicheng--moge-vitl/snapshots/<rev>/model.pt`(符号链接)
  - 实体 blob 大小约 1.2G.
  因此即使 HEAD 超时,通常仍可直接使用本地缓存继续推理.

## 可选缓解方案(不改代码)

- 完全离线(必须已有缓存): `HF_HUB_OFFLINE=1` 或 `TRANSFORMERS_OFFLINE=1`.
- 增大超时(减少误判超时/重试):
  - `HF_HUB_ETAG_TIMEOUT=60`(主要针对 HEAD/ETag).
  - `HF_HUB_DOWNLOAD_TIMEOUT=60`(主要针对 GET 下载).

## 直接证据

- 本机缓存路径(可用 `hf_hub_download(..., local_files_only=True)` 取到):
  - `/root/.cache/huggingface/hub/models--Ruicheng--moge-2-vitl/snapshots/39c4d5e957afe587e04eec59dc2bcc3be5ecd968/model.pt`
- `torch.load(model.pt)["model_config"]` 的关键差异:
  - v2 checkpoint 的 `model_config["encoder"]` 是 dict:
    - `{'backbone': 'dinov2_vitl14', 'intermediate_layers': [5, 11, 17, 23], 'dim_out': 1024}`
  - 并且包含 `neck/points_head/mask_head/scale_head` 等 v2 结构化配置.

## 根因解释(为什么会报 `getattr(): attribute name must be string`)

- 现有推理脚本固定使用:
  - `from moge.model.v1 import MoGeModel`
- 但 v1 的实现假设 `encoder` 是 string,并执行:
  - `hub_loader = getattr(importlib.import_module(\".dinov2.hub.backbones\", __package__), encoder)`
- 当 `encoder` 实际为 dict 时,`getattr(..., encoder)` 直接抛 `TypeError`.

## 修复策略(不改变 MoGe 包本身)

- 加载 MoGe 时自动识别 checkpoint 版本:
  - `model_config.get(\"encoder\")` 是 dict -> 使用 `moge.model.v2.MoGeModel`
  - 否则 -> 使用 `moge.model.v1.MoGeModel`
- 这样同一份代码可以同时兼容:
  - `Ruicheng/moge-vitl`(v1 checkpoint)
  - `Ruicheng/moge-2-vitl`(v2 checkpoint)

# 笔记: 传统 3DGS pipeline vs Lyra(本项目)方法差异

## 结论先讲清楚

- 传统 3DGS: "每个场景单独优化". 你给它多视角 RGB + 相机位姿(常见来自 COLMAP). 它通过可微 splatting 渲染,对每个场景做迭代梯度下降,把一团可渲染的高斯拟合出来.
- Lyra: "先训练一个能直接出高斯的 decoder". 训练数据主要靠"相机可控的视频扩散模型"合成,然后做 self-distillation,把扩散模型里隐含的 3D 先验蒸馏成显式 3DGS. 推理时更像一次前向: 输入是视频 tokenizer latent + 相机条件,输出就是 gaussians.

## 直接证据(代码/文档)

1) 项目定位(README)
- README.md:7: "Feed-forward 3D and 4D scene generation ... trained with synthetic data generated by a camera-controlled video diffusion model"
- README.md:10: "augment the typical RGB decoder with a 3DGS decoder, which is supervised by the output of the RGB decoder"

2) 生成侧: 扩散模型一次采样同时产出 video 和 latents
- cosmos_predict1/diffusion/inference/gen3c_single_image_sdg.py:765-777: `pipeline.generate(... return_latents=True)` 然后 `video, prompt, latents = generated_output`
- cosmos_predict1/diffusion/inference/gen3c_single_image_sdg.py:900-906: `video_latent = latents.detach().float().cpu().numpy(); torch.save(video_latent, latent_save_path)`

3) 重建侧: Lyra 优先吃 `rgb_latents`,没有才走 VAE encode
- sample.py:258-269: `if 'rgb_latents' in batch_test: model_input = batch_test['rgb_latents'] ... else: model_input = encode_multi_view_video(...)`

4) 高斯生成方式: "像素对齐"(沿每条 ray 预测深度再放置高斯中心)
- src/models/recon/model_latent_recon.py:327-351:
  - `distance,... = x.split(...)`
  - `depths = self.dnear * (1 - w) + self.dfar * w`
  - `pos = rays_os + rays_ds * depths`
  - 最后 `gaussians = torch.cat([pos, opacity, scale, rotation, rgbs], dim=-1)`

5) 渲染后端: 用 gsplat 的 rasterization
- src/rendering/gs.py:18: `from gsplat.rendering import rasterization`

## 对比要点(给架构判断用)

- 数据来源:
  - 传统 3DGS 依赖真实多视角采集(或至少真实视频 + SfM).
  - Lyra 的训练不强依赖真实多视角,核心是用扩散模型合成的多视角/轨迹做监督.
- 求解方式:
  - 传统 3DGS 是 per-scene optimization(每个新场景都要重新优化).
  - Lyra 是 feed-forward decoder(同一个 checkpoint 适配很多输入,推理时直接前向输出 gaussians).
- 高斯参数化:
  - 传统 3DGS 的高斯是"自由参数"(位置/尺度/旋转/特征都在优化中漂移),并且常见有 densify/prune.
  - Lyra 这里的高斯更像"像素/patch 对齐的候选集合": 先由 rays + 预测深度给出中心位置,再可选 subsample/prune.
- 颜色/外观:
- 本项目渲染器用的是 `rgbs = gaussians[..., 11:]`(src/rendering/gs.py:40),等价于 SH 的 DC 项(3 通道),没有显式高阶 SH.

# 笔记: `lyra_static_demo_generated` 输出重影的快速诊断(实验0)

## 为什么先做"6 轨迹 -> 1 轨迹"

- 当前的重影更像是"多条轨迹/视角不一致后被融合成厚表面",而不太像单纯分辨率或 intrinsics 的数值错误.

# 笔记: 重影没有随"单轨迹"消失时,下一步怎么做(实验1)

## 现象补充

- 用户反馈: 实验0(只喂 1 条轨迹)后,重影依然很重.
- 这说明问题不只是"多轨迹融合打架".

## 一个很可能的原因: 轨迹本身太激进(尤其是 view idx "5")

我把 `assets/demo/static/diffusion_output_generated/{view}/pose/00172.npz` 里的相机运动做了个快速统计:

- view "5"(clockwise):
  - 相邻帧旋转均值约 0.32°/frame.
  - 相邻帧平移均值约 0.0056(单位是归一化后的世界坐标).
- view "0/1/2":
  - 旋转更小(约 0.05°~0.11°/frame).
  - 平移更小(约 0.0009~0.0019).
- view "3/4"(zoom):
  - 旋转为 0(纯推拉).

直觉上,越"绕"、越"大幅运动",越容易让 diffusion 在细节上漂,最终让高斯像"厚表面"一样叠出重影.

## 我们要做的实验1

目标不是一次就把模型改好.
而是先回答一个更关键的问题:

"换成更温和的轨迹(view 0~4),重影能不能明显变轻?"

做法:

1. 逐个把 `static_view_indices_fixed` 改成 `["0"]..["4"]` 跑一次推理.
2. 用两个指标做横向对比:
   - `PSNR(pred, input)`(越高越好).
   - Laplacian 方差(粗略衡量清晰度,越高通常越锐).

如果实验1能明显改善:

- 结论就是: "轨迹选择"比你改超分/改 intrinsics 更关键.
- 我们可以把默认轨迹从 "5" 换成更稳的那个.

如果实验1仍然都重影:

- 就该进入下一条分支:
  - 更强的 gaussian 过滤(更激进 prune/opacity threshold).
  - 或者走"传统 3DGS per-scene 优化"做后处理 refinement(用 Lyra 输出做初始化).

## 补充: 我加了一个"推理期可调参"的小改良

因为你现在是在做重影对比实验,频繁改 `configs/training/*.yaml` 很容易把自己绕晕.
所以我在 `sample.py` 里加了一个很小的同步逻辑:

- 你可以直接在命令行传:
  - `gaussians_prune_ratio=...`
  - `gaussian_scale_cap=...`
  - `dnear/dfar=...`
  - `pre_sigmoid_distance_shift=...`
- 它会在推理时覆盖到 `main_config`,不用你去改训练配置文件.

我试过把 `gaussians_prune_ratio` 拉到 0.95(只保留 5% 点),
PSNR 会掉得很厉害.
这更像"点太少导致画面变得尖锐但不对".
所以 prune 这条路要谨慎,更建议先做轨迹选择.

# 笔记: 方案3 "相机+高斯联合优化"(refinement)的可落地设计

## 2026-03-05 16:10 UTC

你现在遇到的情况是:

- 单轨迹(单 view)也会出现明显重影.
- 仅靠"轨迹选择"和少量推理期旋钮(例如 `gaussian_scale_cap`)仍不够干净.

这时,最稳的下一步,基本就是把 Lyra 输出当初始化,做一次传统意义上的 per-scene 优化.

我这里把它拆成一个"能写成脚本跑起来"的设计.
并把关键风险点也提前写出来,避免做完一轮才发现方向走不通.

## A. 输入/输出约定(把接口先定死)

### 输入(最小闭环)

1) 初始高斯

- 来源: `outputs/.../gaussians_orig/gaussians_0.ply`
- 说明: 这是已经过激活(opacity sigmoid, scale exp, rgb 归一)后的 14 维高斯,便于直接喂 gsplat 渲染.

2) 监督帧 + 相机

- 建议直接复用 `Provider` 的推理模式产物,避免自己重新对齐 stride/resize:
  - `images_gt`: 对应 `target_index_subsample` 的那批帧(你当前 `rgb_*.mp4` 看到的时间点).
  - `pose_init`: 对应每一帧的 `c2w_init` 或 `w2c_init`.
  - `intrinsics`: 对应 resize/crop 后的内参(和渲染分辨率一致).

### 输出(便于你后续接回 pipeline)

- `refine/gaussians_refined.ply`
- `refine/poses_refined.npz`(只有在启用 pose 优化时才生成)
- `refine/render_refined.mp4`(同一批帧重渲染,用于直观看重影是否下降)
- `refine/metrics.json`(PSNR, sharpness, 可选 SSIM)

## B. 两条实现路线(取决于渲染器对相机是否可导)

### 路线B1: 直接对相机 viewmat 求导(最直觉)

前提:

- `gsplat.rasterization` 对 `viewmats` 有梯度.
- 注意: 官方文档明确写了 `Ks` 目前不可导,所以 intrinsics 默认固定.

优点:

- 实现最简单.
- 不需要在每帧手动把高斯变换到相机坐标系.

风险:

- 如果你用的是 `GaussianRendererDeferred` 这套 deferred BP,它当前 backward 里把相机梯度直接丢掉了(返回 None).
  - 也就是说: 就算 gsplat 支持相机梯度,你也得先绕开 deferred BP,或者把 deferred BP 改造为支持 viewmat 梯度.

### 路线B2(推荐): viewmat 固定为 I,把 pose delta 显式作用在高斯上

动机:

- 不依赖渲染器对 viewmat 的梯度.
- 复用现有 deferred BP/patch BP 的高斯梯度,更省显存.

核心做法:

对每个时间点 t:

1) 用可学习的 `delta_pose_t` 生成 `w2c_t`.
2) 把世界系高斯变到相机系:
   - `means_cam_t = w2c_t * means_world`
   - `quat_cam_t = quat(w2c_t_rot) ⊗ quat_world`(把高斯朝向也一起旋到相机系)
3) 调用 `rasterization` 时:
   - `viewmats = I`
   - 直接用 `means_cam_t / quat_cam_t` 渲染

这样,损失对 `means_cam_t/quat_cam_t` 的梯度会反传到 `delta_pose_t`.
相机优化就成立了.

注意点:

- 这会引入 gauge 自由度(相机和场景一起漂也能拟合).
  - 必须加"锚点"和正则,不然一定跑飞.

## C. 参数化(让优化更稳定)

### C1. Pose delta 用 se(3)

每帧一个 6D:

- `delta_rot_t`: axis-angle(3)
- `delta_trans_t`: translation(3)

用 exp map 得到 4x4 的 `T(delta_t)`.

组合方式建议固定一种,别中途换:

- 推荐: `w2c_t = T(delta_t) @ w2c_init_t`
  - 直觉: 在相机坐标系里做小扰动,更符合"轨迹微调"这件事.

### C2. 高斯参数用"可学习的未激活参数"

- opacity: 用 `opacity_logits` + sigmoid
- scale: 用 `scale_log` + exp,并加一个上限 clamp(对应你之前的 `gaussian_scale_cap` 思路)
- rotation quat: 用 4D raw quat + normalize
- color: 可以先直接优化 RGB(clip 到 [0,1]),不急着上 SH

这样可以避免优化过程中出现负尺度/quat 失范这类直接炸掉渲染的情况.

## D. Loss 与约束(重点,决定你会不会越优化越糊)

### D1. 重建损失(photometric)

建议从简单开始:

- `L_rgb = Charbonnier(render_rgb - gt_rgb)`
  - 比纯 L2 更抗 outlier,对 diffusion 的局部漂移更友好.

可选增强:

- `L_ssim`(低权重,帮助结构对齐,但容易让颜色发灰)

### D2. 相机正则(保证"同一轨迹")

至少要有 3 个:

1) `L_pose_l2 = ||delta_t||^2`(贴近原轨迹)
2) `L_pose_smooth = ||delta_t - delta_{t-1}||^2`(时间平滑)
3) 锚点: 固定 `delta_0 = 0`(直接把第 1 帧锁死,消除整体漂移自由度)

### D3. 高斯正则(防止雾化/飘点)

- `L_scale = mean(scale^2)` 或者对 log-scale 做 L2
- `L_opacity = mean(opacity)`(轻微压一压过多半透明点,减轻"雾感")
- 可选: "位置不乱跑"正则 `||means - means_init||^2`(只在 joint 阶段用很小权重)

## E. 优化日程(建议按阶段跑,别一上来就 joint)

### 阶段E1: pose-only(先把对齐做稳)

- 冻结: 高斯(全部固定)
- 只学: `delta_pose_t`
- 迭代: 200~800 steps(取决于帧数和分辨率)
- 学习率: rotation 小一点,translation 大一点(例如 rot 1e-3, trans 3e-3)

### 阶段E2: joint(小 lr 微调高斯,清掉残留重影)

- 解冻: opacity + color(优先),然后再考虑 means/scale/rot
- 迭代: 1k~5k(按你能接受的时间)
- 学习率: 全部降一档(避免把几何打坏)

### 训练采样(省显存,也更稳)

- 不要每步都吃全分辨率全帧.
- 先做 patch 或 downscale:
  - patch: 例如 352x640 的随机 crop
  - 或者直接把 H/W 降到一半
- 每步随机采样 1~4 帧.
  - diffusion 的某些帧会"漂得更厉害",全喂进去反而逼着模型学妥协,更容易糊.

## F. 我预期你会遇到的坑(提前规避)

1) gauge 问题:

- joint 优化如果没有锚点和 pose 正则,结果一定会"相机和高斯一起漂".
- 指标可能看上去还行,但导出的 pose 会很怪,也不符合"同一轨迹".

2) diffusion 的时序不一致:

- 某些细节在帧间自相矛盾时,优化的最优解就是"厚表面平均掉",看上去反而更糊.
- 所以我更建议:
  - 先 pose-only 解决对齐问题.
  - joint 阶段只小幅微调,并且可以只用一部分关键帧.

3) 直接优化 means/scale 过猛:

- 很容易把局部漂移拟合成"漂移的点云",导致新视角更差.
- 所以 joint 阶段的解冻顺序很重要: 先颜色/opacity,后几何.


## 2026-03-06 00:25 UTC: 重新审视 `joint_refinement_camera_gaussians` 后的判断

### 先说结论

在当前 Lyra 代码和你这批 demo 现象下,**不建议把“pose + gaussian joint refinement”作为默认第一选择**.
更合理的优先级应该改成:

1. `过滤/重加权高斯 + gaussian-only refinement`(首选)
2. `很小幅度的 pose-only 校正`(仅在证据显示是整体相机错位时才启用)
3. `pose + gaussian joint refinement`(最后手段,不是默认路径)

### 为什么我改了之前的判断

#### 1) 当前代码里,直接做相机梯度并不顺

- `src/rendering/gs_deferred.py` 的 `DeferredBP.backward(...)` 最后对 `test_w2cs` 和 `test_intr` 直接返回 `None`.
- 这意味着: 只要沿用项目当前默认的 deferred renderer 路径,你想学相机 pose,梯度链路天然就是断的.
- 即使 `gsplat` 本体支持部分相机相关求导,这里也先被项目自己的 custom autograd 截断了.

#### 2) 现在这个重影,更像“高斯在吸收不一致监督”,不是“纯相机错了”

- `src/models/recon/model_latent_recon.py` 里,高斯中心是 `pos = rays_os + rays_ds * depths`.
- 也就是典型的 pixel-aligned gaussians: 先沿射线预测深度,再直接把高斯放出去.
- 这种结构的优点是快.
- 但代价是: 只要输入视频帧间有一点自相矛盾,它就很容易长出两层近邻表面,最后在渲染里表现成厚表面、双轮廓、半透明重叠.

#### 3) 训练损失本身也偏向“平均掉冲突”

- `src/models/utils/loss.py` 目前是 `MSE + LPIPS + depth + 可选 opacity`.
- 没看到针对 outlier frame / outlier pixel / distractor 区域的 robust masking 或 residual reweighting.
- 所以如果监督彼此冲突,最容易学到的就是“一个能兼顾几边但不够锐”的平均解.

#### 4) 你已经拿到的现象也更支持“先治高斯”

- 单轨迹仍然重影,说明不是多轨迹融合才有的问题.
- `view "3"` 比 `view "5"` 稳,说明相机运动复杂度会放大问题,但不说明根因一定是 pose 本身.
- `gaussian_scale_cap=0.28` 能明显缓解重影,而且我这轮重新统计 `.ply` 也看到它主要是在压缩大尺度高斯尾部:
  - baseline view3: `scale_max > 0.28` 占比约 `0.83%`
  - scale0.28 view3: 该占比直接到 `0%`
  - baseline view5: `scale_max > 0.20` 占比约 `2.90%`,高于 view3 的 `2.31%`
- 这更像“高斯尺度/分布失控会直接放大重影”,而不是“先有 pose 错位,再一切都错”.

### 对旧 spec 的具体修正

旧 spec 里默认节奏是:

1. `pose_only`
2. `joint`

我现在更推荐改成:

1. `gaussian filtering / reweighting`
2. `gaussian-only refinement`
3. `pose-only diagnostic refinement`(可选)
4. `joint refinement`(只在前 3 步都不够时)

### 更合适的落地方向

#### 方向 A: 过滤/重加权高斯 + gaussian-only refinement(推荐默认)

目标不是一上来“让 pose 也能学”,而是先减少那些明显在制造双轮廓的高斯.

建议拆成两小步:

- A1. 预过滤:
  - 继续保留 `gaussian_scale_cap`
  - 再增加按 `opacity / scale / depth stability / render residual` 的过滤或重权重
- A2. 短迭代 refinement:
  - 先只优化 `opacity + color + scale`
  - `means/rotation` 先冻结,或者只允许非常小的受限更新
  - loss 改成 robust photometric,至少用 `Charbonnier/L1`,并对高残差区域降权

#### 方向 B: pose-only,但只作为诊断工具

什么时候才值得动 pose:

- 如果 residual 看起来是“整张图同方向错一点”,例如所有边缘都往同一方向偏.
- 而不是局部这里一层、那里一层的厚表面.

在那种情况下,pose-only 才可能真有效.
否则它更像是在替 diffusion 时序不一致背锅.

#### 方向 C: joint refinement 只做最后兜底

只有当:

- 先做了 robust 的 gaussian-only,仍有明显系统性错位;
- 并且你接受轨迹有极小幅度修正;
- 并且我们愿意改 renderer 链路或走“显式把 pose 作用到高斯”这条更重的路;

这时才值得上 joint.

### 一个更实用的判断树

```text
单轨迹重影
  |
  +-- scale cap 收紧后明显变好 --> 先治高斯分布,不是先治 pose
  |
  +-- 残差是局部双轮廓/半透明叠层 --> gaussian-only + robust masking
  |
  +-- 残差是整图一致性偏移 --> 再考虑 tiny pose-only
  |
  +-- 前两步都不够 --> 最后才上 joint
```

### 外部方法上的参考启发

- `Robust Gaussian Splatting` 这类工作,本质是在 blur / pose noise / color inconsistency 下做鲁棒建模,说明“先把不稳监督当 outlier 处理”是合理方向,不是旁门左道.
- `Robust 3D Gaussian Splatting for Novel View Synthesis in Presence of Distractors` 这类工作,核心也是通过 residual/mask 忽略污染监督,这和你当前“扩散视频局部不一致”这个问题在方法论上很接近.


## 2026-03-06 00:44 UTC: `RobustSplat` 论文 + 仓库 对 Lyra 的启发

### 论文 / 仓库核心点

结合 arXiv `2506.02751` 和仓库 `fcyycf/RobustSplat`,它的主线非常清楚:

1. **Delayed Gaussian Growth**
   - 先让静态结构收敛,再允许 densification(split / clone).
   - 论文动机是: 标准 3DGS 的 densification 太早发生时,会把 transient/disturbance 也长成新高斯.
   - 仓库默认参数也明显体现了这个设计:
     - `densify_from_iter = 10000`
     - `densify_until_iter = 20000`
     - `bootstrap_iter = 20000`

2. **Scale-cascaded Mask Bootstrapping**
   - 不是一开始就用高分辨率 mask 硬训.
   - 前期先用更低分辨率 DINO 特征做更稳的 coarse mask.
   - 后期再切换到 fine mask,提高边界精度.
   - 仓库实现位置:
     - `train.py`
     - `utils/mask_utils.py`

3. **它解决的是“动态/瞬态干扰污染静态 3DGS”**
   - 论文讲 transient objects.
   - 但方法论上,这和 Lyra 当前遇到的“扩散视频局部不一致 / 帧间自相矛盾”非常像.
   - 两者共同点都是: 有些监督不该被当成稳定几何去拟合.

### 和 Lyra 的关系: 哪些是同构问题

Lyra 当前不是传统 per-scene 3DGS 训练,而是 feed-forward 预测 gaussians.
但你现在看到的重影,方法论上和 RobustSplat 处理的问题很接近:

- RobustSplat 里的“transient object”
- Lyra 里的“diffusion 帧间漂移 / 双轮廓 / 时序不一致细节”

都属于:

- **不稳定监督进入 3D 表达后,被平均成脏几何**

所以它再次强化了一个判断:

> 对 Lyra 来说,更应该先做“监督鲁棒化 + 高斯鲁棒化”,而不是优先做相机联合优化.

### 仓库实现上最值得借鉴的层

#### A. Mask / Residual Reweighting 层

这是最值得借鉴的一层.

仓库里做法是:

- 用 DINOv2 提特征.
- 用一个很小的 MLP 预测 mask.
- 用 residual histogram 生成上下界 mask(`lower_mask`, `upper_mask`).
- 用 feature cosine + residual interval 一起训练 mask.
- 正式重建 loss 只在 `loss_mask` 区域上算.

这层的意义不是“分割人/车”.
而是:

- **找出当前不值得相信的监督区域,别让它们主导高斯更新**.

这和 Lyra 很契合.
因为你当前最像是某些局部区域在不同帧里不一致,而不是整帧都坏了.

#### B. Delayed Densification / Delayed Geometry Freedom

RobustSplat 的论文重点之一是:

- 不要在几何还没稳定时就大量 split/clone.

Lyra 虽然没有传统 3DGS 那种训练期 densification 流程.
但它有一个非常相似的“几何自由度释放”问题:

- 一旦太早允许 `means/scale/rotation` 大幅自由更新,它也会把不一致监督吸收到几何里.

所以在 Lyra refinement 里可以对应成:

- **先只动 `opacity/color/有限 scale`,延后 `means/rotation` 的解冻**.

这其实就是 RobustSplat 思想在 Lyra 语境下的等价翻译.

#### C. Camera optimization 不是它的主角

这个也很关键.

从论文与仓库看,RobustSplat 的主要武器不是 camera refinement.
它依赖的是:

- 固定相机
- 鲁棒 mask
- 延迟 densification

这再次支持我们前面已经得出的结论:

- **joint camera + gaussians 不该是 Lyra 当前的默认第一选择**.

### 对 Lyra 可迁移性评估

#### 高可迁移

1. residual-based mask / weight map
2. coarse-to-fine 的 mask bootstrapping 思路
3. 先 appearance/opacity,后 geometry 的 staged refinement
4. 对高残差区域降权,而不是强行拟合

#### 中等可迁移

1. DINO 特征监督
   - 可以做,但代价较重.
   - 对 inference-time 单场景 refinement 来说未必是第一步.
2. 历史残差直方图阈值
   - 可以借鉴成 EMA residual quantile,比较实用.

#### 低可迁移 / 不适合直接照搬

1. 传统 3DGS 的 densify-and-clone 机制
   - Lyra 当前不是这条训练范式.
   - 直接照搬会很重,也会把任务改大.
2. 把 transient segmentation 完整复制过来
   - 对你的问题来说,不一定需要先做“语义级瞬态分割”.
   - 更可能只需要 residual-driven soft weighting 就够了.

### 这会不会改变我们之前的优先级判断?

会,但方向不是推翻,而是**进一步坐实**.

新的把握更强了:

1. `过滤/重加权高斯 + robust loss` 依然应该排第一.
2. `gaussian-only refinement` 依然应该早于 pose-only / joint.
3. `tiny pose-only` 只在看到整图一致错位证据时才介入.
4. `joint refinement` 继续保留为最后兜底.

### 如果把 RobustSplat 思想翻译成 Lyra 版,最像下面这个流程

```text
Lyra 输出初始 gaussians
  |
  +-- 渲染当前 target frames
  |
  +-- 计算 per-pixel residual / perceptual residual
  |
  +-- 生成 soft weight map
  |      前期: 粗分辨率,稳一点
  |      后期: 细分辨率,精一点
  |
  +-- 用 weight map 做 robust photometric loss
  |
  +-- phase 1: 只优化 opacity / color / limited scale
  |
  +-- phase 2: 若仍有局部厚表面,再小幅开放 geometry
  |
  +-- 仍存在全局一致偏移? 再考虑 tiny pose-only
```

## 2026-03-06 01:05 UTC: V2 规格落盘

- 已新建 `specs/joint_refinement_camera_gaussians_v2.md`.
- V2 不再把 `pose_only -> joint` 当默认主线.
- 新的推荐顺序明确为:
  1. residual-driven weighting
  2. gaussian-only appearance-first refinement
  3. limited geometry refinement
  4. tiny pose-only diagnostic
  5. joint fallback
- V2 里保留了两个档位:
  - 档位 A: 最佳方案,带 coarse-to-fine weighting 与完整阶段化流程
  - 档位 B: 先能用方案,只做 RGB residual + Stage 2A
- 这版规格本质上是把 `RobustSplat` 的方法论翻译成了适合 Lyra 的后处理路线,而不是直接照搬传统 3DGS 训练范式.


## 2026-03-06 01:24 UTC: `tttLRM` 对当前 Lyra V2 的帮助边界

### 先说总判断

`tttLRM` **有启发,但不是当前问题的直接解法**.

它最有价值的地方在于:

- 提供了一种比普通 transformer / feed-forward LRM 更强的**长上下文吸收机制**.
- 允许把更多 posed views 通过 TTT layer 的 fast weights 压进隐式 3D latent,再解码成 GS.
- 支持 autoregressive / online progressive reconstruction.

但它**不直接解决**你现在最核心的问题:

- 单轨迹视频内部的时序不一致
- 局部双轮廓
- 厚表面
- 高斯被不稳定监督污染

换句话说:

> `tttLRM` 更像“更强的前馈重建底座”.
> `RobustSplat` 更像“更适合当前 ghosting 问题的方法论”.

### 我为什么这么判断

#### 1) 它解决的主问题是 long-context / autoregressive scaling

从论文摘要和项目页,tttLRM 的关键词一直是:

- long-context
- autoregressive 3D reconstruction
- linear computational complexity
- online learning variant

也就是说,它首先在意的是:

- 输入视角很多时,怎么高效吸收
- 怎么流式地继续吸收更多视角
- 怎么在长序列下继续做 feed-forward 3DGS reconstruction

这和你当前的第一痛点不是一回事.
你当前的第一痛点是**重影和监督不稳**,不是上下文不够长.

#### 2) 它的 TTT 不是“给现有 Lyra 加一个小后处理”

仓库代码看下来,`tttLRM` 的 TTT 是模型主干结构的一部分.
不是后面额外接一个小优化器.

关键点:

- `model/model.py` 里主模型类就是 `tttLRM`.
- `block_config` 里直接用了 `model.lact_ttt.FastWeightGluMLPMultihead`.
- `ttt_scan` 支持 `full` 和 `ar` 两种扫描模式.
- `ar` 模式下通过 `miniupdate_views` 控制每次用多少视角块更新 fast weights.

这说明它是**架构级替换**,不是推理后补一层 refinement.

#### 3) 它仍然假设 posed multi-view 输入是可信的

`tttLRM` 的输入是 posed images + rays.
代码里非常明确:

- 先根据 `fxfycxcy` 和 `c2w` 计算 rays.
- 再把 `ray_o / ray_d / cross(ray_o, ray_d) / image` 拼成输入与 query.
- 再解码成 GS.

它对“输入足够多,视角足够长”非常友好.
但对“输入局部自相矛盾”这件事,我没有看到它像 `RobustSplat` 那样专门做了 residual masking / robust weighting.

#### 4) 它的 loss 也不是面向 ghosting 的专门设计

在 `model/model.py` 里能看到的 loss 还是这一类:

- L2 / PSNR
- perceptual
- opacity
- depth

这和 Lyra 当前的 loss 家族其实是同类型的.
也就是说,**它没有直接给出一套专门抑制双轮廓 / 厚表面的监督鲁棒化机制**.

### 它对当前 V2 哪些地方有启发

#### A. 有帮助: 作为“未来更强前端”的方向

如果未来我们不满足于当前 Lyra 的 backbone,想升级成更强的 large reconstruction model,`tttLRM` 很有参考价值.

尤其是这几个点:

1. **长上下文吸收**
   - 如果未来你想喂更多轨迹 / 更长单轨迹,它的线性复杂度与 AR chunking 很有吸引力.

2. **online progressive update**
   - 如果未来希望“先有粗 GS,再随着更多帧流入逐步更新”,它给了一个架构方向.

3. **query/update 解耦**
   - 它把 input tokens 用来更新 fast weights,再用 virtual queries 解码 GS.
   - 这类 query/update 分离思路,对将来做 streaming reconstruction 很有意思.

#### B. 中等帮助: 对 V2 的阶段设计有间接支持

虽然它不能直接替代 V2,但它会加强一个直觉:

- **不要一次把所有问题都丢给固定前馈结构去硬吃**.
- 允许“输入被吸收 -> 内部状态更新 -> 再输出”的过程,通常比纯单步更强.

翻译到 Lyra V2 里,这更支持:

- phase 式 refinement
- 分阶段释放自由度
- 先诊断,后更新

#### C. 直接帮助不大: 当前 ghosting 修复

对你眼前这个问题,它没有 `RobustSplat` 那么直接.

因为你现在缺的优先不是:

- 更长上下文
- 更强 AR 模型
- 更大的重建底座

你现在更缺的是:

- residual-driven weighting
- robust supervision
- 高斯尾部治理
- 分阶段 geometry release

### 所以它会不会改变 V2?

**不会改主线,但会影响“远期路线图”.**

#### 不会改变的部分

V2 现在的主线仍然成立:

1. residual-driven weighting
2. gaussian-only appearance-first refinement
3. limited geometry refinement
4. tiny pose-only diagnostic
5. joint fallback

#### 会补强的部分

它会让我更愿意在 V2 后面加一条“远期架构升级方向”:

- 如果后续发现问题不仅是 ghosting,还包括“更多视角喂不进去”或“多段轨迹需要流式融合”,那时可以考虑 `tttLRM/Long-LRM` 这条架构升级线.

### 适配性判断

#### 高适配: 未来架构升级参考

- 长上下文多视角重建
- 流式 / 分块更新
- autoregressive chunk processing

#### 低适配: 当前最小改动修重影

- 直接接到现有 Lyra V2 后处理上
- 快速降低单轨迹双轮廓
- 低侵入修当前 ghosting

### 最后结论

如果问题是:

- “当前 Lyra 的单轨迹重影怎么先修好?”

那优先级仍然是:

1. `RobustSplat` 思想
2. 我们已经写好的 V2
3. `tttLRM` 暂不进入当前实施主线

如果问题变成:

- “后面要不要换一个更强的 reconstruction backbone,支持更长上下文 / 多段轨迹 / 在线吸收更多输入?”

那 `tttLRM` 就会变得非常值得跟进.


# 笔记: `LongSplat` 对当前 Lyra 重影问题的帮助边界

## 时间

- 2026-03-06 04:49 UTC

## 先说结论

`LongSplat` **有帮助,但不是当前问题的一线解法**.

它更适合解决的是:

- 无位姿视频怎么逐步注册进场景
- 长视频怎么做增量式 3DGS 优化
- pose drift / 新帧插入失败 / 长序列显存与优化窗口管理

它不直接解决的是:

- 扩散视频内部的不一致监督
- 单轨迹中的局部双轮廓 / 厚表面 / ghosting
- 如何按 residual 对不可靠监督降权

所以对当前 Lyra 的判断是:

- **不改 V2 主线**
- **可以作为未来 "无位姿长视频 / 多段轨迹 / 外部视频 pose 不可靠" 路线的参考实现**

## 一手证据

### 1) README 的定位就很清楚: 它首先是 unposed + long videos

`/tmp/LongSplat/README.md` 里直接写了:

- 标题: `Robust Unposed 3D Gaussian Splatting for Casual Long Videos`
- Overview: 它是一个面向 casually captured long videos 的 `unposed 3D Gaussian Splatting framework`
- 关键词明确落在:
  - `incremental joint optimization`
  - `pose estimation with MASt3R`
  - `adaptive octree anchoring`

这说明它的主问题是:

- 输入视频没有可靠 pose
- 视频很长,不能一次性全吃
- 需要边注册边优化,同时控制内存和锚点规模

而不是“已经有一条轨迹,但内容重影很重”.

### 2) 它的训练主循环,核心在 "注册新帧 -> 局部优化 -> 全局优化 -> 长程 refinement"

从 `/tmp/LongSplat/train.py` 可以看到非常典型的增量式流程:

- 初始化阶段: 先用少量初始帧做优化
- 新帧进入时:
  - 先用 `cv2.solvePnPRansac(...)` 做 pose 初始化
  - 再做 pose-only refinement
- 然后做:
  - `local_iter`
  - `global_iter`
  - 最后 `post_iter` 长程 refinement

而且配置参数也围绕这个流程组织:

- `/tmp/LongSplat/arguments/__init__.py`
  - `pose_iteration = 200`
  - `local_iter = 400`
  - `global_iter = 900`
  - `window_size = 5`
  - `post_iter = 20000`

这进一步说明 LongSplat 解决的是:

- 新帧怎么并进来
- 局部窗口怎么滚动
- 全局怎么再统一
- 长视频怎么持续优化

### 3) 它确实用了遮挡与可见性机制,但目标更像注册稳健性,不是监督鲁棒化

`/tmp/LongSplat/utils/graphics_utils.py` 有 `get_occlusion_mask(...)`.
`/tmp/LongSplat/train.py` 和 `/tmp/LongSplat/utils/pose_utils.py` 多处用它配合 pose estimation.

这个机制有价值.
但它的核心作用更接近:

- 提高配准和相机 refinement 的稳定性
- 避免把明显不可见区域拿去做错误约束

它**不是** `RobustSplat` 那种:

- 以 residual / feature inconsistency 为中心
- 生成 soft weight map
- 对 photometric supervision 做逐像素鲁棒降权
- 延迟 geometry 自由度释放

所以它和我们现在 V2 里的 `residual-driven weighting` 不是同一层东西.

### 4) README 自己也承认: 需要足够 overlap,还要尽量避免 dynamic objects

`/tmp/LongSplat/README.md` 的 "Run on your own video" 里明确建议:

- 避免 dynamic objects
- 保证 frames 之间 sufficient camera overlap

这反而说明它的强项不是去硬扛不一致内容.
它更像是在一个仍然相对 "静态 + 可重叠" 的视频上,把无 pose 的长序列重建做稳.

而你当前 Lyra 遇到的问题,恰恰已经有点超出这条假设:

- 单轨迹本身就可能有 diffusion 造成的局部内容不稳定
- 相同结构在不同时刻不完全一致
- pixel-aligned gaussians 容易把这种不一致平均成厚表面

## 和当前 Lyra V2 的关系

### 不改变主线

当前 V2 主线仍然成立:

1. residual-driven weighting
2. gaussian-only appearance-first refinement
3. limited geometry refinement
4. tiny pose-only diagnostic
5. joint fallback

原因很简单:

- 当前 Lyra 的主痛点,仍然是监督不一致和高斯尾部造成的 ghosting
- 不是 "pose 一定先错了"
- 也不是 "长视频吞不下"

### 哪些点有借鉴价值

`LongSplat` 仍然有几块值得我们吸收,但更像 future route:

1. **增量注册框架**
   - 如果以后外部视频没有可靠 pose,或者轨迹来自多段视频拼接,它的 PnP + pose refinement 流程很值得借鉴.

2. **局部窗口 + 全局统一**
   - 如果以后不是 1 条短轨迹,而是更长的视频或多段片段,它的 local/global schedule 很有价值.

3. **可见性驱动的窗口移动**
   - `train.py` 里会根据 visibility ratio 推进窗口.
   - 这个对长视频的效率和稳定性都很实用.

4. **anchor/octree 结构**
   - 如果未来 Lyra 要从当前高斯表示进一步升级到更适合大场景/长序列的结构,LongSplat 的 anchor 管理值得单独研究.

### 哪些点不该现在就搬进来

1. **不要把 pose-first 当成当前默认解法**
   - LongSplat 的 pose-first 是因为它本来就在解 unposed video.
   - Lyra 当前不是这个前提.

2. **不要把局部/全局优化 schedule 当成直接治 ghosting 的核心手段**
   - 它会帮助长序列一致性.
   - 但不会自动解决 diffusion 监督污染.

3. **不要误以为 occlusion mask = robust supervision**
   - 两者不是一回事.
   - 前者主要筛不可见约束.
   - 后者要处理“可见但不可信”的监督.

## 最终判断

### 对当前问题

如果问题是:

- “怎么先把当前 `lyra_static_demo_generated` 里的单轨迹重影降下去?”

那么优先级仍然是:

1. `RobustSplat` 方法论
2. 当前 `joint_refinement_camera_gaussians_v2`
3. `LongSplat` 暂不进入当前主实施路线

### 对未来扩展

如果后面问题变成下面这些,LongSplat 就会显著升值:

- 外部视频没有可靠 pose
- 视频长度明显更长
- 需要多段轨迹逐步并入同一场景
- 出现 pose drift / 新帧注册难 / 长程一致性问题
- 需要更省内存的增量式 3DGS 架构

到那时,更合理的做法不是直接替换 V2.
而是:

- 保留 V2 作为“鲁棒监督 + 去重影”层
- 再叠加一条 `LongSplat-style unposed incremental reconstruction` 路线

## 一句话总结

`RobustSplat` 更像是在治你现在的 "监督不稳导致重影".
`LongSplat` 更像是在治 "无 pose 长视频怎么稳地重建".

所以它**有帮助**,但**不是当前这道题最该先做的那一把刀**.

## 2026-03-06 05:37 UTC: 探索“超分视频能否改善 `sample.py` 高斯清晰度”的真实边界

### 1) 当前主链为什么不适合靠“直接提高视频分辨率”来改善高斯

先看 `sample.py` 的输入分支:

- `sample.py:277-287`
  - 有 `rgb_latents` 时,直接 `model_input = batch_test['rgb_latents']`
  - 只有没有 latent 时,才会走 `images_input_vae` -> `encode_multi_view_video(...)`

再看 provider 的行为:

- `src/models/data/provider.py:74`
  - `self.load_latents = self.opt.load_latents and self.has_latents`
- `configs/training/default.yaml`
  - 默认 `load_latents: true`
- `src/models/data/registry.py`
  - `lyra_static_demo_generated_one` / `lyra_static_demo_generated` 都是 `has_latents: true`

这几条合起来的意思是:

- 默认 demo 链路会优先读 `latent/*.pkl`,而不是重新根据 mp4 编码.
- 所以你就算把 `rgb/*.mp4` 替换成超分版,只要 `latent/*.pkl` 还在,`sample.py` 看的主要还是旧 latent.

### 2) 就算强行走 RGB 分支,超分细节也不会自然保留下来

`src/models/data/provider.py:_preprocess(...)` 做了:

- `rgbs, depths, shift, scale, flip_flag = self.image_transform.preprocess_images(...)`
- 然后把 intrinsics 一起按同一个 `shift + scale` 重算

`src/models/utils/data.py:preprocess_images(...)` 明确会:

- 先 center crop
- 再 resize 到 `sample_size`

而当前 demo / 主训练档位是:

- `configs/demo/lyra_static.yaml`
  - 继承 `configs/training/default.yaml` 和 `configs/training/3dgs_res_704_1280_views_121_2.yaml`
- `configs/training/default.yaml`
  - `img_size: [704, 1280]`

这意味着:

- 即便你关掉 `load_latents`,把超分视频当 RGB 输入,
- 它进入模型前依然会被裁回当前工作分辨率档位.

所以“超分过的视频”并不会自动把更高频的信息完整传递到当前 feed-forward 高斯生成器里.

### 3) 如果想真正换分辨率,那已经不是“小修小补”,而是进入新的分辨率 regime

还有两个代码信号说明当前系统对主分辨率档位是有默认预期的:

- `src/models/utils/model.py:get_encoder_chunk_size(...)`
  - 对 `T=121` 只特判了 `H=704`
- `src/rendering/gs_deferred.py:136`
  - 渲染分辨率直接取 `self.opt.img_size`

所以如果你想让主链真正吃更高分辨率:

- 不只是换视频文件.
- 还要连 `img_size`、intrinsics 缩放、latent 编码尺寸、显存预算、甚至 checkpoint 泛化边界一起重审.

结论不是“理论上绝对不可能”.
而是:

- **它不属于当前现成可行路径**.
- **一旦硬做,就已经接近一条新的分辨率支持工程**.

### 4) 更合理的方向: 把 `sample.py` 当初始化器,在末尾接高斯 refinement

这个方向反而更顺:

- `sample.py` 已经能导出 `gaussians_*.ply`
- 现有 spec V2 也已经把 per-scene refinement 路线拆清楚

并且 `src/rendering/gs_deferred.py:106` 明确表明:

- backward 返回 `xyz/feature/scale/rotation/opacity` 的梯度
- 对相机相关输入直接返回 `None`

这说明:

- 现在最自然的是 `gaussian-only refinement`
- 不适合把“相机也一起学”当第一步

### 5) “超分对等视频做监督”在什么前提下成立

要让这个方向真正成立,至少要满足:

1. 视频内容严格对等
   - 不是“长得更像”,而是同一帧同一结构同一时序.
   - 如果超分模型引入时序抖动或 hallucination,refinement 会把错误细节烙进高斯.
2. intrinsics 要按倍率同步缩放
   - 如果只是 2x 超分,理论上 `fx/fy/cx/cy` 也跟着乘 2.
   - 前提是 crop / aspect ratio 完全不变.
3. refinement 渲染端要支持更高 supervision 分辨率
   - 当前 renderer 的输出尺寸取决于 `opt.img_size`
   - 所以这个阶段最好独立成 post stage,由它自己控制 patch / 高分辨率渲染

### 6) 两条方案的真实取舍

#### 方向 A: 最佳方案

- 保持 `sample.py` 主链不变
- 新增 `post_refine_sr` 阶段
- 输入:
  - `gaussians_init_ply`
  - 原 pose / intrinsics
  - 超分后的对等视频
- 优化:
  - 先 `appearance-first gaussian-only refinement`
  - 再决定要不要放开 limited geometry
- 监督:
  - patch-based 高分辨率 RGB / perceptual / residual weighting

这条路的优点是:

- 不污染 feed-forward 主链
- 真正能利用高分辨率 supervision
- 和现有 `joint_refinement_camera_gaussians_v2` 完全同方向

#### 方向 B: 先能用方案

- 不做真正的高分辨率监督
- 继续在 704x1280 档位做:
  - `gaussian_scale_cap`
  - residual weighting
  - gaussian-only refinement

这条路更快,但它吃不到超分带来的额外监督细节.

### 7) 当前最稳的判断

一句话总结:

- **“先超分,再直接喂回 `sample.py` 主链”不属于当前靠谱路径.**
- **“`sample.py` 跑完后,再拿超分对等视频做高斯 refinement”是合理且更值得实现的路径.**

如果后面真要开工,我会更推荐把它设计成:

- 一个独立的 post-refinement stage
- 而不是把优化循环硬塞进 `sample.py` 主推理循环里

因为从架构本质上说,这两者已经不是同一种范式:

- `sample.py` 是 feed-forward inference
- 末尾补一轮是 test-time per-scene optimization


## 2026-03-06 06:08 UTC: 评估 `tttLRM` 与 `Long-LRM` 对 Lyra 的可借鉴性

### 先给一句总判断

这两个项目都值得学.
但它们适合 Lyra 的层次不一样:

- `Long-LRM` 更适合借“长序列 feed-forward 高斯重建”的工程套路.
- `tttLRM` 更适合借“在线更新 / test-time state adaptation / streaming reconstruction”的方法论.

如果目标是当前 Lyra 这条 `sample.py -> gaussians -> refinement` 路线:

- **短中期优先学 `Long-LRM`**
- **中长期再评估 `tttLRM`**

### 一, `Long-LRM` 的核心价值是什么

从论文和项目页看, `Long-LRM` 的主问题非常明确:

- 不是“如何做 streaming online adaptation”
- 而是“如何把 wide-coverage, 32-view, 高分辨率的 feed-forward GS 重建做得既快又稳”

核心设计落点:

1. 长序列骨干
   - `Mamba2 + Transformer` 混合骨干
   - 目标是把 32 张 960x540 图像带来的超长 token 序列吃下去
2. token merging
   - 不是改任务定义
   - 是在中途主动缩 token,控显存和时延
3. Gaussian pruning
   - 训练时就用 opacity 正则把高斯变稀疏
   - 推理时直接 prune,减少渲染和反传成本
4. post-prediction optimization
   - 先 feed-forward 预测高斯
   - 再对输入视角做几步高斯参数优化

这一点和 Lyra 当前状态特别接近:

- 我们现在也已经在往“先 feed-forward,后 per-scene refinement”走.
- `Long-LRM` 论文里已经明确做了一个轻量后优化版本.

### 二, `Long-LRM` 哪些点最适合借到 Lyra

#### 1) 最值得直接借: post-prediction optimization 的思路

Long-LRM 论文明确写了:

- 先预测一版高斯
- 再在输入视角渲染
- 用输入图像和渲染图的 MSE 反传到高斯参数
- 第一版只改 position 和 color,不动 opacity / scale / rotation

这对 Lyra 特别有价值,因为:

- 它和我们当前的 `post-refinement` 方向完全同类
- 它比“直接把超分视频硬回灌主链”自然得多

但 Lyra 不建议原样照搬.
更适合改成:

- `appearance-first gaussian-only refinement`
- 先动 `color / opacity / limited scale`
- 再视情况放开受限几何

原因是 Lyra 当前主要症状更偏:

- 厚表面
- 双轮廓
- 半透明叠层

而不只是“位置不够准”.

#### 2) 很值得借: opacity regularization + pruning 的方法论

Long-LRM 训练里:

- 用 depth regularization 稳定位置
- 用 opacity regularization 鼓励稀疏
- 最终把低 opacity 高斯 prune 掉

这和我们现在观察到的现象高度契合:

- `gaussian_scale_cap=0.28` 之所以有效,本质上就在压大尺度尾部和厚层
- 如果后续 refinement 要更稳,opacity / visibility / scale 的联合约束非常值得借

可迁移到 Lyra 的方式:

- refinement 阶段加入更显式的 opacity 稀疏约束
- diagnostics 中加入更明确的 `opacity_lowconf_ratio` / `scale_tail_ratio`
- 在 refinement 前或中途引入更保守的高斯筛除策略

#### 3) 结构上值得学: 2D GS / 高斯变体兼容思路

Long-LRM 论文和项目页都强调:

- 同一套大框架可以兼容 `3D GS` 和 `2D GS`
- 2D GS 对 geometry 有帮助

这对 Lyra 的启发不是“马上切 2D GS”.
而是:

- 当前 refinement / diagnostics / adapter 层最好不要把高斯表示写死得太死
- 后面如果要探索 geometry 更稳的高斯参数化,Long-LRM 这条兼容思路很有参考价值

#### 4) 长期值得学: 长序列骨干与 token 预算控制

如果未来 Lyra 要从当前几帧/单轨迹,走向:

- 更长视频
- 更大分辨率
- 更多输入视角

那 `Long-LRM` 的这些设计会很有价值:

- hybrid `Mamba2 + Transformer`
- token merging
- 分阶段分辨率 curriculum

但这类借用属于“重做骨干网络”的级别.
它不是当前最优先的事情.

### 三, `Long-LRM` 的代码能不能直接借

可以参考学习.
但要分层看:

#### 可以直接参考的部分

- repo 结构
  - `configs/`, `data/`, `model/`, `main.py`, `inference.py`
- custom data / camera normalization 的处理思路
- post-prediction optimization 的实现组织
- 训练 / 推理入口与 config 组织方式

#### 不能指望直接拿来贴进 Lyra 的部分

- 主干网络实现本身
  - 它依赖自己的 token 化、view 采样、camera normalization、训练数据契约
- 训练循环
  - 它解决的是 posed multi-view image -> GS 的直接回归
  - Lyra 现在解决的是 latent / camera conditioning 下的特定 decoder 链路

#### 代码层现实判断

`arthurhero/Long-LRM` 是 self-reimplemented 版本.
优点是:

- 训练和评估管线都在
- 对学习工程组织很有帮助

限制是:

- 不应把它当“官方逐文件一比一真值”
- 更适合作为“设计与实现参考”

### 四, `tttLRM` 的核心价值是什么

`tttLRM` 的重点和 `Long-LRM` 不同.
它真正新的地方不在“更强一点的高斯头”,而在:

- 把输入图像压进 TTT / LaCT 的 fast weights
- 把 fast weights 视为一种可在线更新的隐式 3D 记忆
- 再用 virtual tokens 去 query 这块记忆,解码成 3DGS 等显式表示

也就是说,它的核心不是普通 feed-forward encoder.
而是:

- `stateful memory`
- `online update`
- `causal / autoregressive reconstruction`

### 五, `tttLRM` 哪些点值得 Lyra 学

#### 1) 最值得学的是“在线 refinement / streaming update”的方法论

论文里明确说:

- 它支持 streaming 输入
- 每来一小批视图,就更新 fast weights
- 然后立即输出当前 3DGS

这对 Lyra 的价值主要在未来:

- 如果以后我们要做“边进视频边修高斯”
- 或者“长视频逐段增量融合”
- 或者“test-time adaptation 不是几步 Adam,而是更像带记忆的状态更新”

那 `tttLRM` 的框架非常值得研究

#### 2) 值得学的是“预训练 NVS 知识迁移到显式 3D”的观点

论文强调:

- 从 novel view synthesis 预训练迁移到显式 3D 建模,会让收敛更快、结果更好

这点和 Lyra 的哲学很接近.
因为 Lyra 本身就很依赖上游视频 / latent 先验.

它给我们的启发是:

- 以后如果要做更强的 post-refinement / adapter
- 可以考虑让 refinement 不只是从随机参数开始
- 而是显式复用已有的隐式或生成模型先验

#### 3) 值得学的是 sequence parallel 的工程思路

`tttLRM` 仓库 README 明确提到:

- 用 `sp_size` 做 sequence parallel
- 输入视图和生成高斯会分摊到多卡

如果以后 Lyra 要做:

- 更长上下文
- 更高分辨率 supervision
- 多卡 patch-based refinement

sequence-parallel 的思路值得借鉴.

### 六, `tttLRM` 哪些地方不适合当前 Lyra 直接照搬

#### 1) 它太偏“新主干”,不是“给现有链路补丁”

`tttLRM` 不是在现有 LRM 上加一个小头.
它换的是“记忆机制 + 推理范式”.

要把它真正并入 Lyra,大概率要重做:

- 主干 token 处理
- 状态更新逻辑
- query / decoder 接口
- 训练方式

这已经不是当前的“小步改良”.

#### 2) 它对当前问题不是最短路径

我们当前最现实的问题是:

- 单轨迹重影
- 超分监督怎么补到后处理里
- 怎么把现有高斯再优化得更干净

这些问题,`tttLRM` 不是最短路径解法.
因为它更像:

- “下一代更强重建器怎么设计”

而不是:

- “现有高斯怎么在末尾补几步优化”

#### 3) 代码可借性弱于 `Long-LRM`

从仓库 README 看:

- 它公开了 inference
- 训练代码暂未完全提供
- 并且 README 还明确说训练可以“结合 LongLRM 和他们的 inference / sequence parallel 部分”自己做

所以代码层面:

- `tttLRM` 更适合学 ideas 和推理组织
- 不适合期待“直接拿来复用一大段训练/改造代码”

### 七, 回到 Lyra, 应该怎么排序

#### 当前阶段(最务实)

优先借:

1. `Long-LRM` 的 post-prediction optimization
2. `Long-LRM` 的 opacity regularization / pruning 思路
3. `Long-LRM` 的 camera normalization / custom inference 组织

原因:

- 这些和 Lyra 当前 V2 refinement 最贴
- 改造成本最低
- 收益最直接

#### 下一阶段(如果要做更长序列或增量融合)

继续借:

1. `Long-LRM` 的 token budget 管理
2. `LongSplat` 风格的增量注册 / 长视频组织
3. `tttLRM` 的在线状态更新思路

#### 更远阶段(如果要做新主干)

重点研究:

1. `tttLRM` 的 LaCT / fast-weight memory
2. implicit memory -> explicit GS 的 query 解码
3. streaming autoregressive reconstruction

### 八, 一句话决策建议

如果你问“今天我们最应该学哪一个来推进 Lyra 现在这条线”:

- 答案是 `Long-LRM`.

如果你问“哪一个更像下一代架构方向,值得做中长期研究储备”:

- 答案是 `tttLRM`.

- 我们最值得继续学 `Long-LRM` 的地方,不是“要不要抄一套新主干”.
- 而是“如何把现有这些半到位的设计,变成更完整、更系统的训练/推理/后处理链”.

### 二, 迁移清单总览

```text
Long-LRM / tttLRM -> Lyra

低侵入:
  A. post-prediction optimization
  B. opacity / pruning / diagnostics 强化
  C. token budget / subsampling 策略校准

中侵入:
  D. 更系统的 token merging
  E. 更自由但受限的 gaussian offset / geometry schedule
  F. 多视图 / 长序列输入 curriculum

高侵入:
  G. LaCT / fast-weight memory
  H. streaming autoregressive reconstruction
  I. sequence-parallel + online state update 主干
```

### 三, 现在最值得做的 3 个模块

#### A. `Long-LRM` 的 post-prediction optimization

##### 可借内容

- 先前向得到高斯
- 再用输入视角监督做少量优化

##### 对应 Lyra 文件

- 初始化输出:
  - `sample.py`
  - `src/models/recon/model_latent_recon.py`
- 渲染:
  - `src/rendering/gs_deferred.py`
  - `src/rendering/gs_deferred_patch.py`
- 规划中的落地点:
  - `specs/joint_refinement_camera_gaussians_v2.md`
  - 目标脚手架:
    - `scripts/refine_robust_v2.py`
    - `src/refinement_v2/*`

##### 为什么它是第一优先级

因为它和当前用户问题直接相连:

- 超分视频无法自然回灌 feed-forward 主链
- 但可以作为 `sample.py` 输出高斯之后的监督信号

##### 迁移时怎么改造

不建议按 Long-LRM 原样做“先 position + color”.
Lyra 更适合:

1. Stage 2A:
   - 先 `opacity + color + limited scale`
2. Stage 2B:
   - 再视情况放开 `rotation + limited means`

##### 侵入级别

- 低侵入

##### 风险

- 如果直接全量高分辨率整帧渲染,显存会炸
- 应优先走 patch-based supervision

#### B. `Long-LRM` 的 opacity / pruning 体系

##### 可借内容

- opacity regularization
- 训练/推理 pruning 一致性
- 更系统的“高斯稀疏性”指标

##### 对应 Lyra 文件

- 训练 loss:
  - `src/models/utils/loss.py`
- 高斯后剪枝:
  - `src/models/recon/model_latent_recon.py:372+`
  - `src/rendering/gs.py`
  - `src/rendering/gs_deferred.py`
  - `src/models/utils/render.py:204+`
- 配置:
  - `configs/training/default.yaml`
  - `configs/training/3dgs_res_704_1280_views_121_2.yaml`

##### 为什么它对当前问题有用

当前 Lyra 的视觉问题很像:

- 高斯太厚
- 半透明叠层太多
- scale tail 和 opacity tail 一起导致 ghosting

所以这块比“直接上更复杂 pose 优化”更对症.

##### 可以怎么落地

1. 先把训练期已有的 `lambda_opacity + gaussians_prune_ratio` 再系统化
2. 把 diagnostics 补齐:
   - `opacity_lowconf_ratio`
   - `scale_tail_ratio`
   - `opacity_histogram`
3. refinement 阶段再加:
   - `L_opacity_sparse`
   - 更保守的 prune / freeze 规则

##### 侵入级别

- 低侵入

#### C. `Long-LRM` 的 token budget 管理

##### 可借内容

- token merging 思想
- 更明确的序列预算控制
- 在长上下文下先降冗余,再做重建

##### 对应 Lyra 文件

- `src/models/utils/render.py:127+`
- `src/models/utils/token_pruning.py`
- `configs/training/default.yaml`
- `src/models/recon/model_latent_recon.py:250+`

##### 当前为什么不是第一刀

因为 Lyra 现在的最紧急问题不是“上下文太长导致完全吃不下”.
而是:

- 现有结果重影
- 现有高斯后处理还没形成闭环

所以它更适合在:

- refinement 跑起来之后
- 或者未来输入视角数和分辨率继续上去时

再往前推.

##### 侵入级别

- 低到中侵入

### 四, 中侵入层: 值得参考,但不是立即开工

#### D. 更系统的 token merging

Lyra 现在已有:

- `sub_sample_gaussians_type`
- `sub_sample_gaussians_type_tokens`
- learned mask

但还没有形成像 `Long-LRM` 那样明确的“中途降 token 预算”的整体策略.

这条线可以做,但前提是:

- 先证明当前 ghosting 主要不是 supervision 问题
- 或者未来上下文确实要继续变长

#### E. 更自由但受限的 geometry schedule

Lyra 现在有:

- `gaussians_predict_offset`

但默认关着.
这说明仓库已经给未来更自由的位置微调留了口子.

如果以后要借 `Long-LRM` 那种更主动的 geometry 微调:

- 优先接在 refinement 的 Stage 2B
- 不要先放进 feed-forward 主链

#### F. 多视图 / 长序列 curriculum

`Long-LRM` 的强项之一是:

- 它天然就是更大视角覆盖和更长输入序列

Lyra 如果以后要从当前 demo 扩到:

- 更多输入视角
- 更高 coverage
- 更长视频片段

那它的数据组织和 curriculum 会很值得学.
但这已经超出当前单轨迹去重影的主问题.

### 五, 高侵入层: 现在不要急着做

#### G. `tttLRM` 的 LaCT / fast-weight memory

这块最有研究价值.
但它本质上会改:

- 主干状态表示
- 上下文压缩方式
- query 解码方式

所以不适合作为当前 Lyra V2 的下一步.

#### H. streaming autoregressive reconstruction

适合未来场景:

- 外部长视频
- 逐段输入
- 增量融合

它更像 `LongSplat V3 + tttLRM` 的交叉方向.

#### I. sequence parallel + online state update

如果以后要把 Lyra 做成:

- 边读长视频边更新
- 多卡长上下文在线推理

这条线很强.
但现在显然过早.

### 六, 一个更实用的“现在/以后”排序

#### 现在就能推进的

1. `Long-LRM` 的 post-prediction optimization
2. `Long-LRM` 的 opacity / prune / diagnostics 强化
3. 把 Lyra 当前 V2 refinement 做成最小闭环

#### 等最小闭环跑通再做的

1. token merging / learned pruning 更系统化
2. 受限 geometry offset
3. 更长输入序列与更高 coverage

#### 作为下一代架构储备的

1. `tttLRM` 的 LaCT
2. fast-weight memory
3. streaming autoregressive reconstruction

### 七, 直接给出一份“建议行动单”

如果未来从 explore 切到实现,最合理的顺序是:

1. 先做 `refine_robust_v2` 最小版
   - 不碰主干
   - 不碰 pose
   - 只做 gaussian-only
2. 在这个最小版里明确吸收 `Long-LRM` 的两件事
   - post-pred optimization 的组织方式
   - opacity / prune 的规则与指标
3. 等这条线稳定后,再考虑:
   - learned token merging
   - geometry offset
4. 最后再决定是否单开新分支研究:
   - `tttLRM-style Lyra`

### 八, 一句话总结

对 Lyra 当前阶段来说:

- `Long-LRM` 更像“应该立刻借的工程增量”.
- `tttLRM` 更像“应该先记住,但别急着实现的新范式”.
